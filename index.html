Workflows & Tools

The right mix is important. Tools can foster great Culture. Workflows can improve efficiency.

Workflows:

Since we are technical folks here I will use technical terms to describe the workflows.

Pipeline 1: Client/Sales Team -> PM -> Engineering -> QA -> Ops -> Users

Pipeline 2: Users -> Support -> PM -> Engineering -> QA -> Ops -> Users

- PM / Engineering / QA / Ops / Support

PM Group: System Analysts, Project Managers, Product Managers, Program Managers

  • PM Group takes input from the customer/sales and defines the work to be done
  • Different PM mehodologies - different ways to do that (more or less customer involvment, waterfall, iterative)
  • Different tools: Gantt charts, Kanban boards, Milestones
  • Developers are like computers, if you don't define something, they will always produce the wrong thing

Engineering: Software Developers, Software Architects, DBAs or as Balmer would put it: Developers, Developers, Developers, Developers, ...

  • The Real Heroesâ„¢ here
  • Do all the work, get none of the credit

QA: QA Engineer (sad truth about our job - we create things that replace people)

  • Get the 'finished' work from the developers and prove to them that it is far from finished
  • Do all the work, get none of the credit

Ops: Network Engineer, DevOps Engineer

  • Keep it running
